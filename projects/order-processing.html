Multithreaded Order Processing System

Producer-consumer model for processing retail orders with thread-safe operations and structured logging

C++ pthreads Semaphores Mutexes POSIX Threads

Project Overview

Developed a sophisticated multithreaded order processing system that simulates a retail environment where multiple producers generate customer orders and multiple consumers process them concurrently. The system demonstrates advanced understanding of parallel programming concepts and thread synchronization.

The implementation uses the classic producer-consumer pattern with a shared buffer, ensuring thread-safe access through careful use of POSIX threads, semaphores, and mutexes. This project showcases my ability to build robust concurrent systems that handle real-world scenarios like inventory management and order fulfillment.

Completed in May 2025 as part of my coursework in parallel computing, this project required deep understanding of operating system primitives, race condition prevention, and deadlock avoidance.

Key Features

Producer-Consumer Model

Implemented multiple producer threads generating orders and consumer threads processing them from a shared circular buffer with proper synchronization.

Thread-Safe Operations

Used semaphores and mutexes to ensure proper synchronization, preventing race conditions and ensuring data integrity across concurrent operations.

Circular Buffer Management

Designed efficient circular indexing for the shared buffer with producer-end signaling to coordinate thread communication and prevent buffer overflow.

Structured Logging System

Built comprehensive logging to track customer transactions, order status, and thread activities for debugging and performance analysis.

Inventory Persistence

Integrated file I/O for loading initial inventory, tracking changes during order processing, and regenerating formatted output for audit trails.

Performance Optimization

Optimized thread scheduling and synchronization overhead to maximize throughput while maintaining correctness and avoiding deadlocks.

Technical Implementation

The system architecture revolves around POSIX threads (pthreads) with careful synchronization using semaphores for buffer counting and mutexes for critical section protection. Producer threads generate orders at varying rates and place them in a bounded circular buffer, while consumer threads retrieve and process orders, updating inventory accordingly.

Key challenges included preventing race conditions when multiple threads access shared data structures, implementing proper signaling for producer termination, and ensuring fair scheduling among consumers. I used counting semaphores to track empty and full buffer slots, and a mutex to protect the critical sections where the buffer index is modified.

The logging system captures detailed transaction information including thread IDs, timestamps, order details, and inventory changes. File parsing handles initial inventory loading, and output regeneration creates formatted reports showing final inventory state and transaction history.

Results & Learning Outcomes

Zero Race Conditions

Achieved complete thread safety with no race conditions or data corruption across thousands of test orders through rigorous synchronization.

Deadlock-Free Design

Carefully ordered lock acquisition and proper semaphore usage ensured the system never deadlocked, even under high thread contention.

Scalable Architecture

System scales efficiently with configurable numbers of producer and consumer threads, maintaining correctness at any scale.

Deep Systems Understanding

Gained hands-on experience with low-level threading primitives, synchronization mechanisms, and concurrent programming best practices.
